# Вопросы с se.ifmo.ru

## Цели и задачи интеграционного тестирования. Расположение фазы интеграционного тестирования в последовательности тестов; предшествующие и последующие виды тестирования ПО.

### Цели интеграционного тестирования

**Цель интеграционного тестирования** — проверить, что отдельные модули, компоненты или подсистемы корректно работают вместе после объединения.

### Задачи интеграционного тестирования:

проверка интерфейсов и взаимодействия между модулями;
   * вызовы API, сообщения
   * сетевое взаимодействие
   * проверка связи между **бизнес-логикой и пользовательским интерфейсом**;
   * ...

### Расположение фазы интеграционного тестирования

```
 требования  <------------------бизнес требования, валидация ---------------------------------------> Приемочное тестирование
 
    Анализ <--------------------Требования к ПО, Валидация, верификация -------------------> Системное тестирование
    
      Архитектурное <---------- Требования к интерфейсам ----------------------> Интеграционное тестирование
      проектирование            Верификация
      
            Детальное  <------------- Тестовый случаи -------------------> Модульное тестирование
            проектирование            Верификация
          
                                           
                                       Разработка
```

### предшествующие и последующие виды тестирования ПО

Предшествующий вид тестирования — **модульное тестирование**.
На этом этапе проверяют отдельные функции, классы, модули изолированно.

Последующий вид тестирования — **системное тестирование**.
Здесь уже проверяют всю систему целиком на соответствие требованиям.

Итого цепочка обычно такая: Модульное → Интеграционное → Системное → Приёмочное

## Алгоритм интеграционного тестирования.

0. **Определить интегрируемые модули**, **понять интерфейсы взаимодействия между ними**

    * Сначала выбирают, какие компоненты будут объединяться и проверяться вместе.

1. **Определить стратегию интеграции**

   * сверху вниз (down);
   * снизу вверх (up);
   * функциональная (end-to-end, по одной функции);
   * ядро/backbone (похож на постепенная интеграция/incremental integration): предварительно интегрируйте основные компоненты, которые создают основной поток, а затем постепенно расширяйте.
   * большрй взрыв/big bang: интегрируйте все сразу, а затем протестируйте.

2. **Выбрать порядок объединения модулей**

   * последовательность важна;
   * выбор зависит от архитектуры ПО;
   * чем больше одновременно интегрируемых частей, тем выше сложность.

3. **Подготовить план интеграционных тестов**

   * для каждого интерфейса составить короткий тест-план;
   * определить, какие взаимодействия проверяются:

     * API,
     * сообщения между компонентами,
     * БД,
     * UI ↔ логика,
     * сетевые, аппаратные, локальные, инфраструктурные интерфейсы.

4. **Подготовить тестовую среду**

   * подключить готовые компоненты;
   * для отсутствующих частей использовать:

     * **stub/заглушки** — при стратегии сверху вниз;
     * **driver/драйверы** — при стратегии снизу вверх.

5. **Пошагово интегрировать модули**

   * объединить первую группу компонентов;
   * выполнить тесты их взаимодействия;
   * по мере готовности подключать остальные компоненты.

6. **Провести проверки взаимодействия**

   * корректность обмена данными;
   * правильность вызовов и ответов;
   * корректность форматов, параметров, последовательности вызовов;
   * обработка ошибок на границах модулей.

7. **Зафиксировать и исправить дефекты**

   * выявить ошибки интерфейсов и взаимодействия;
   * после исправления повторить тесты.

8. **Расширять интеграцию до полной сборки**

   * добавлять новые компоненты;
   * каждый раз повторять цикл: интеграция → тестирование → исправление.

9. **Проверить систему в собранном виде**

   * убедиться, что все подсистемы работают совместно;
   * при необходимости проверить и нефункциональные характеристики.

Кратко в виде схемы:

**Выбор стратегии → План интерфейсов → Подготовка заглушек/драйверов → Поэтапная сборка модулей → Проверка взаимодействия → Исправление дефектов → Повторение до полной интеграции**

Могу ещё сделать **очень короткий экзаменационный ответ** на 5–6 строк.


## Концепции и подходы, используемые при реализации интеграционного тестирования.

### Сверху вниз

Интеграционное тестирование __сверху вниз__  - сначала тестируют только самый верхний управляющий уровень системы, без модулей более низкого уровня. Затем постепенно с более высокоуровневыми модулями интегрируются более низкоуровневые.

__Преимущества:__

- Легко обнаружить неисправности или ошибки в работе системы.
- В первую очередь проверяются важные модули, а лишь потом модули нижнего порядка.
- По сравнению с другими подходами, время на тестирование интеграции очень коротко.

__Недостатки:__

- Если в модули нижнего уровня заложена важная логика, она не может быть протестирована в первую очередь
- Использование «заглушек»становится обязательным на всех последующих проектах.

### Снизу вверх

Интеграционное тестирование __снизу вверх__ (сначала в лабе протестировали синус и ln, потом sin, cos, lnб потом все тригонометрические и логарифмические функции и т.д) - подход подразумевает проверку низкоуровневых систем для начала: вместе и по отдельности. Другими словами процесс тестирования начинается с внутреннего уровня и постепенно доходит до наиболее критичных позиций.

__Преимущества:__

- Если определенный модуль перестает функционировать, его ошибка может быть сразу же исправлена.
- Требуется минимальное время на идентификацию и устранение ошибок.

__Недостатки:__

- Общее время проверки всех модулей довольно долгое
- Если ПО содержит много небольших модулей мелкого уровня, которые очень сложны в своей имплементации, то для завершения процесса тестирования может потребоваться больше времени.

### Большой взрыв

Тестирование методом __большого взрыва__ -  созданные и запрограммированные модули и системные компоненты соединены между собой. При объединении эти модули тестируются как единое целое. После проведения юнит-тестов, модули также проверяются вместе, еще до образования целостной программной системы.

__Преимущества:__

- Удобен  при тестировании небольших систем.
- Быстрое нахождение ошибок

__Недостатки:__

- Так как модули завязаны на одной системе, порой очень трудно найти источник дефектов.
- Если в системе используется много модулей, может уйти достаточно времени, чтобы пересмотреть все реализованные функциональности.

## Программные продукты, используемые для реализации интеграционного тестирования. Использование JUnit для интеграционных тестов.

Mockito - фреймворк с поддержкой макетирования для unit-тестов.

```java
import org.junit.jupiter.api.Test;
import java.util.List;

import static org.mockito.Mockito.*;

class MockitoExampleTest {
    @Test
    void demoMockito() {
        List<String> list = mock(List.class);

        when(list.get(0)).thenReturn("hello");

        assert "hello".equals(list.get(0));
        verify(list).get(0);
    }
}
```

Greenmail — для тестирования электронной почты, который поддерживает SMTP (Simple Mail Transfer Protocol - send email to mail server), POP3(Post Office Protocol version 3/ take email from server) и IMAP (Internet Message Access Protocol) с поддержкой SSL-соединения. 

```java
import com.icegreen.greenmail.util.GreenMail;
import com.icegreen.greenmail.util.ServerSetupTest;
import com.icegreen.greenmail.util.GreenMailUtil;
import org.junit.jupiter.api.*;

class GreenMailExampleTest {
    GreenMail greenMail = new GreenMail(ServerSetupTest.SMTP);

    @BeforeEach
    void start() {
        greenMail.start();
    }

    @AfterEach
    void stop() {
        greenMail.stop();
    }

    @Test
    void demoGreenMail() throws Exception {
        GreenMailUtil.sendTextEmailTest(
            "to@localhost",
            "from@localhost",
            "subject",
            "body"
        );

        assert greenMail.getReceivedMessages().length == 1;
    }
}
```

MockFtpServer — библиотека, которая предоставляет две разные реализации FTP (File Transfer Protocol/ transfer file through internet)-сервера («заглушка» и «обманка»), которые можно использовать для тестирования различных сценариев.

# Другие вопросы
## В чем отличие между Dummy, Fake object, Stub, Mock, Spy?

* Dummy - это пустой объект для передачи параметра. В тесте он не влияется. Например, в конструкторе, которому нужен еще один параметр, он поставлен
* Fake object - это упрощенная рабочая реализация компонента. Например, вместо настояющей БД используем HashMap для хранения данных в in-memory памяти.
* Stub - это тестовой компонент. Вместо реального компонента он возвращает заранее подготовленный ответ
* Mock - это тестовой компонент, похожий на Stub, но еще можем проверять, был ли вызован метод, сколько раз вызовано, ...
* Spy - это внешняя оболочка реального компонента, обычно реально работающего. Мы можем следовать за его вызовами или подменить часть поведения

## Нужно ли тестировать заглушки?

Простые заглушки обычно отдельно не тестируют, поскольку они используются только для симуляции поведения зависимости в рамках теста. 

Однако если заглушка представляет собой сложную или переиспользуемую реализацию, например fake object или эмулятор сервиса, ее необходимо проверять, так как ошибки в ней могут исказить результаты тестирования.

## В чем суть стратегии Big Bang? Почему она считается наиболее рискованной?

Big Bang — это стратегия, при которой все модули интегрируются сразу и затем тестируются как единое целое.
Она считается наиболее рискованной, потому что ошибки обнаруживаются поздно, их много одновременно, и очень трудно определить, какой именно модуль или интерфейс вызвал проблему.

## Как концепция контрактного тестирования (Consumer-Driven Contracts, CDC) соотносится с классическим интеграционным тестированием? Может ли CDC полностью заменить интеграционные тесты?

### концепция CDC

- **CDC (Consumer-Driven Contracts)**:
  - проверяет **контракт** между consumer и provider
  - подтверждает, что provider отдает именно то, что ожидает consumer

### Связь между CDC и интеграционным тестированием

- CDC **дополняет** интеграционные тесты
- CDC проверяет **совместимость контракта**
- Интеграционные тесты проверяют **реальную совместную работу системы**
- CDC обычно:
  - быстрее
  - проще в запуске
  - раньше находит ошибки
- Интеграционные тесты обычно:
  - тяжелее
  - медленнее
  - но ближе к реальным условиям

### Может ли CDC полностью заменить интеграционные тесты?

- **Нет, обычно не может**
- CDC заменяет только часть проверок
- CDC не проверяет всю систему в реальной среде
- поэтому на практике:
  - CDC используют для быстрой проверки контрактов
  - интеграционные тесты оставляют для критичных реальных сценариев


## Какие виды дефектов невозможно обнаружить при модульном тестировании, но можно выявить только на этапе интеграции? Приведите 3 примера.

Да. Модульное тестирование проверяет отдельный компонент изолированно, поэтому оно **не выявляет ошибки взаимодействия между модулями**. Такие дефекты обычно обнаруживаются только на этапе интеграции.

- **1. Несовместимость интерфейсов между модулями**
Один модуль передает данные в одном формате, а другой ожидает в другом.
Например, сервис заказов отправляет дату в формате `DD-MM-YYYY`, а сервис оплаты ожидает `YYYY-MM-DD`.

- **2. Ошибки передачи и обработки данных между компонентами**
Отдельно каждый модуль работает правильно, но при совместной работе данные теряются, искажаются или неверно интерпретируются.
Например, модуль скидок передает итоговую сумму без НДС, а модуль оплаты считает, что НДС уже включен.

- **3. Ошибки последовательности вызовов и взаимодействия**
Каждый модуль по отдельности проходит unit-тесты, но в реальной цепочке вызовов нарушается порядок работы.
Например, модуль уведомлений пытается отправить сообщение до того, как модуль заказов завершил сохранение заказа в базе.


## Как изменится подход к созданию заглушек, если модуль, который мы замещаем, является stateful (хранит состояние между вызовами), а не stateless? Как симулировать состояние в заглушке?

Если замещаемый модуль **stateful**, то простой заглушки вида **“на один вызов вернуть заранее заданный ответ”** уже недостаточно.

### В чем меняется подход

Для **stateless**-модуля заглушка обычно очень простая:

* на вход X → вернуть Y
* каждый вызов независим от предыдущего

Для **stateful**-модуля заглушка должна учитывать, что:

* результат текущего вызова зависит от **предыдущих вызовов**
* внутри есть **внутреннее состояние**
* возможны разные ответы в зависимости от этапа работы

То есть заглушка должна не просто возвращать данные, а **моделировать жизненный цикл состояния**.

---

### Что нужно делать в заглушке

#### 1. Хранить внутреннее состояние

Например:

* флаг
* счетчик вызовов
* коллекцию данных
* текущий статус объекта
* карту `id -> state`

Пример:

* первый вызов создает задачу
* второй возвращает статус `IN_PROGRESS`
* третий — `DONE`

---

#### 2. Менять состояние после вызовов

Заглушка должна не только читать state, но и **обновлять его**.

Например:

* `save(user)` сохраняет пользователя во внутренний `Map`
* `delete(id)` удаляет его из `Map`
* `findById(id)` читает уже измененное состояние

---

#### 3. Поддерживать сценарии переходов состояния

Иногда нужно моделировать не просто хранение, а **state machine**:

* `NEW -> PROCESSING -> DONE`
* `CREATED -> PAID -> SHIPPED`
* `LOCKED / UNLOCKED`

Тогда заглушка должна проверять допустимость переходов.

---

## Как симулировать состояние

Самый частый способ — хранить данные в памяти:

* переменные
* `List`
* `Map`
* `Set`
* счетчики
* enum-статусы

---

## Простой пример

Допустим, реальный модуль хранит пользователей.

```java id="bzyrkn"
class FakeUserRepository {
    private final Map<Long, String> storage = new HashMap<>();

    public void save(Long id, String name) {
        storage.put(id, name);
    }

    public String findById(Long id) {
        return storage.get(id);
    }

    public void delete(Long id) {
        storage.remove(id);
    }
}
```

Это уже stateful-заглушка, потому что:

* после `save()` состояние меняется
* `findById()` зависит от предыдущих вызовов
* `delete()` тоже влияет на будущие результаты

---

## Пример со статусом

```java id="r6p37c"
class FakePaymentService {
    private String status = "NEW";

    public void pay() {
        status = "PAID";
    }

    public String getStatus() {
        return status;
    }
}
```

Здесь ответ `getStatus()` зависит от того, вызывали ли раньше `pay()`.

---

## Пример со счетчиком вызовов

Иногда состояние — это просто номер шага:

```java id="jmbygk"
class FakeJobService {
    private int pollCount = 0;

    public String getJobStatus() {
        pollCount++;
        if (pollCount == 1) return "QUEUED";
        if (pollCount == 2) return "RUNNING";
        return "DONE";
    }
}
```

Так можно симулировать long-running process.

---

## Что важно учесть

### 1. Заглушка становится сложнее

Для stateful-модуля это уже часто не просто stub, а скорее:

* **fake**
* или тестовый эмулятор

---

### 2. Нужно уметь сбрасывать состояние между тестами

Иначе тесты будут влиять друг на друга.

Например:

```java id="ji8krm"
fakeRepository.clear();
```

или создавать новый объект на каждый тест.

---

### 3. Нужно моделировать только значимое поведение

Не стоит копировать весь production-код.
Заглушка должна быть:

* проще реального модуля
* предсказуемой
* детерминированной
* достаточной для тестов

---

### 4. Иногда полезно задавать начальное состояние

Например:

* пустое хранилище
* уже существующий заказ
* счет со статусом `BLOCKED`

Это позволяет тестировать разные сценарии.

---

## Главное отличие от stateless-заглушки

### Stateless stub:

* просто возвращает заранее заданный ответ

### Stateful stub:

* хранит данные между вызовами
* меняет внутреннее состояние
* выдает ответ в зависимости от истории взаимодействия

---

## Короткий ответ для экзамена

Если замещаемый модуль является **stateful**, заглушка должна моделировать не только вход и выход, но и **внутреннее состояние**, которое сохраняется между вызовами. Для этого в заглушке используют внутренние переменные, коллекции, счетчики или статусы, изменяющиеся после каждого вызова. Таким образом симулируется поведение реального модуля, у которого результат операций зависит от предыдущих действий.
