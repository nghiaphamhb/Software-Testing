## 1. Функциональное тестирование. Основные понятия, способы организации и решаемые задачи

### 1.1. Что такое функциональное тестирование

Функциональное тестирование - это вид тестирования, при котором проверяется, выполняет ли система свои функции в соответствии с требованиями. Основное внимание уделяется не внутренней реализации программы, а ее внешнему поведению: какие входные данные подаются, какие действия выполняет пользователь и какой результат получает система.

Иначе говоря, функциональное тестирование отвечает на вопрос:  
`Работает ли приложение так, как ожидает пользователь и как описано в требованиях?`

Для веб-приложений это особенно важно, потому что пользователь взаимодействует с интерфейсом напрямую: открывает страницы, нажимает кнопки, вводит данные, получает сообщения об ошибках, переходит между разделами.

### 1.2. Основные понятия

#### Функция
Это конкретная возможность системы, например:
- авторизация пользователя;
- поиск данных;
- переход по разделам сайта;
- редактирование профиля;
- установка отметки `Star` у репозитория.

#### Требование
Это описание того, как функция должна работать. Именно требования являются основой для проверки.

#### Use case (прецедент использования)
Это описание цели пользователя и его взаимодействия с системой.  
Пример: `Пользователь входит в систему`, `Пользователь ищет репозиторий`, `Пользователь редактирует профиль`.

#### Тестовый сценарий
Это более конкретное описание проверяемого потока внутри use case.  
Например, для use case `Authentication` сценариями могут быть:
- открытие страницы логина;
- вход с неверными данными;
- вход с корректными данными;
- выход из системы.

#### Тест-кейс
Это уже детализированная проверка с условиями, шагами и ожидаемым результатом.

#### Checklist
Это перечень того, что обязательно должно быть проверено. В нем обычно фиксируют краткие пункты без длинного описания шагов.

#### Ожидаемый результат
Это поведение системы, которое считается правильным. Во время тестирования фактический результат сравнивается именно с ним.

### 1.3. Что проверяет функциональное тестирование

Функциональное тестирование проверяет:
- корректность бизнес-логики;
- правильность обработки пользовательского ввода;
- корректность переходов между страницами;
- появление нужных элементов интерфейса;
- корректность сообщений об ошибках;
- выполнение операций при валидных и невалидных данных.

Оно не отвечает за:
- производительность;
- нагрузку;
- безопасность на глубоком техническом уровне;
- удобство интерфейса как отдельную дисциплину.

### 1.4. Способы организации функционального тестирования

Функциональное тестирование можно организовывать по-разному.

#### 1. Ручное тестирование
Тестировщик сам открывает приложение и выполняет шаги вручную.  
Плюсы:
- просто начать;
- удобно для первичной проверки.  
Минусы:
- долго;
- сложно повторять одинаковые проверки много раз;
- выше риск человеческой ошибки.

#### 2. Автоматизированное тестирование
Сценарии оформляются в виде программного кода, который сам выполняет действия в браузере и проверяет результат.  
Плюсы:
- быстрый повторный запуск;
- удобно проверять регресс;
- можно включить в CI/CD.  
Минусы:
- нужна настройка среды;
- необходимо поддерживать тесты при изменении UI.

#### 3. Организация по use case
Это удобный способ для учебного проекта и для отчетности.  
Сначала выделяются основные прецеденты использования, затем для каждого use case строятся:
- checklist;
- test scenarios;
- автоматизированные тесты.

Именно такой подход считается наиболее логичным, потому что тестовое покрытие строится от пользовательских целей.

### 1.5. Какие задачи решает функциональное тестирование

Функциональное тестирование решает следующие задачи:
- подтверждает, что система соответствует требованиям;
- выявляет ошибки в логике работы;
- подтверждает корректность пользовательских сценариев;
- проверяет, что изменения не ломают уже существующую функциональность;
- дает основание для вывода о качестве интерфейса и стабильности основных функций.

### 1.6. Пример на основе данной лабораторной работы

В данной работе тестировался интерфейс GitHub. Функциональное тестирование здесь включает, например:
- проверку аутентификации;
- проверку поиска;
- проверку открытия публичного репозитория;
- проверку переходов по вкладкам `Issues` и `Pull requests`;
- проверку `Star/Unstar`;
- проверку редактирования собственного профиля;
- проверку `Follow/Unfollow`.

Таким образом, тестирование было построено от набора use case, а затем реализовано в виде автоматизированных тестов на Selenium.

### 1.7. Краткий вывод

Функциональное тестирование - это проверка того, что система правильно реализует ожидаемые функции. Оно строится на use case, сценариях и ожидаемых результатах, а его главная задача - подтвердить корректность работы системы с точки зрения пользователя.

## 2. Система Selenium. Архитектура, принципы написания сценариев, способы доступа к элементам страницы

### 2.1. Что такое Selenium

Selenium - это набор инструментов для автоматизации тестирования веб-приложений. Он позволяет программно управлять браузером, выполнять действия пользователя и проверять результат работы интерфейса.

Selenium широко используется для UI-тестирования, потому что умеет:
- открывать страницы;
- находить элементы;
- нажимать кнопки;
- вводить текст;
- получать данные со страницы;
- работать с вкладками, окнами, alert, frame;
- запускать тесты в разных браузерах.

### 2.2. Основные компоненты Selenium

#### Selenium IDE
Инструмент для записи действий пользователя в браузере по принципу `record and playback`.  
Подходит для:
- быстрого прототипирования;
- создания шаблонов сценариев;
- демонстрации простых последовательностей действий.

#### Selenium WebDriver
Основной современный API Selenium для управления браузером.  
Именно WebDriver используется в большинстве реальных проектов и в данной лабораторной работе.

#### Selenium Grid
Компонент для распределенного и параллельного запуска тестов.  
Позволяет выполнять сценарии:
- на разных браузерах;
- на разных операционных системах;
- на разных машинах.

### 2.3. Архитектура Selenium

В упрощенном виде архитектура выглядит так:

1. Тестовый код отправляет команды.
2. WebDriver API передает эти команды драйверу браузера.
3. Драйвер браузера взаимодействует с реальным браузером.
4. Браузер выполняет действия и возвращает результат.

Например, в Java:
- `WebDriver` - это интерфейс;
- `ChromeDriver` - реализация для Chrome;
- `FirefoxDriver` - реализация для Firefox.

То есть тест не работает с браузером напрямую, а через слой драйвера.

### 2.4. Базовые принципы написания сценариев

При написании Selenium-сценариев важно соблюдать несколько принципов.

#### 1. Сценарий должен быть воспроизводимым
Тест должен запускаться повторно с тем же ожидаемым результатом.

#### 2. Сценарий должен быть привязан к пользовательскому потоку
Тест нужно строить не от случайных кликов, а от use case.

#### 3. Нужно разделять структуру страницы и тестовую логику
Обычно для этого используют Page Object Model.

#### 4. Нужно учитывать асинхронность веб-интерфейса
Элементы могут появляться не сразу, поэтому сценарий должен корректно ждать нужного состояния.

#### 5. Локаторы должны быть устойчивыми
Нельзя опираться на хрупкие абсолютные пути в DOM или на нестабильные идентификаторы, если страница динамическая.

### 2.5. Page Object Model

Page Object Model - это подход, при котором каждая страница или функциональный блок приложения оформляется отдельным классом.

Например:
- `LandingPage`
- `LoginPage`
- `HomePage`
- `RepositoryPage`
- `ProfilePage`

Такой подход дает преимущества:
- код становится чище;
- локаторы собраны в одном месте;
- тестовые методы становятся понятнее;
- при изменении UI легче обновлять проект.

Вместо длинных цепочек действий прямо в тесте:

```java
driver.findElement(...).click();
driver.findElement(...).sendKeys(...);
```

лучше вызывать методы страницы:

```java
landingPage.clickSignIn();
loginPage.login(username, password);
```

### 2.6. Ожидания и синхронизация

В веб-тестах одна из ключевых проблем - несинхронность интерфейса.  
Страница может открыться, но элемент еще не готов к использованию.

Поэтому в Selenium применяются:

#### Implicit wait
Глобальное ожидание при поиске элемента.

#### Explicit wait
Ожидание конкретного условия через `WebDriverWait` и `ExpectedConditions`.

Примеры условий:
- элемент видим;
- элемент кликабелен;
- текст появился;
- URL изменился;
- элемент исчез.

Explicit wait обычно предпочтительнее, потому что он точнее и лучше управляется.

### 2.7. Способы доступа к элементам страницы

В Selenium элемент на странице обычно представляется объектом `WebElement`.

Основные способы поиска:
- `By.id`
- `By.name`
- `By.className`
- `By.tagName`
- `By.linkText`
- `By.partialLinkText`
- `By.cssSelector`
- `By.xpath`

В данной лабораторной работе по условию основной способ - `XPath`.

### 2.8. `WebElement` и основные методы

`WebElement` - это объект, представляющий HTML-элемент на странице.

Основные методы:
- `click()`
- `sendKeys()`
- `clear()`
- `getText()`
- `isDisplayed()`
- `isEnabled()`
- `isSelected()`
- `findElement()`
- `findElements()`

Эти методы позволяют как выполнять действия, так и получать информацию о состоянии элемента.

### 2.9. `@FindBy`, `PageFactory` и локаторы

В Page Object часто используется аннотация `@FindBy`, которая указывает, как искать элемент.

Пример:

```java
@FindBy(xpath = "//input[@name='login']")
private WebElement usernameInput;
```

Это делает код страницы более структурированным и позволяет хранить локаторы отдельно от тестового сценария.

`PageFactory` используется для инициализации таких элементов внутри page object.

### 2.10. Архитектура проекта в этой лабораторной работе

В текущем проекте используется следующая структура:

- `SeleniumConfigurator` - инициализация `WebDriver`, `WebDriverWait`, `JavascriptExecutor`;
- `pages/*` - page object классы;
- `src/test/java/*` - тесты по use case;
- `Allure` - генерация отчета;
- `.env` - хранение учетных данных и параметров запуска.

Это соответствует хорошей практике: инфраструктура, page object и тесты отделены друг от друга.

### 2.11. Краткий вывод

Selenium - это система автоматизации веб-тестирования, основанная на управлении браузером через WebDriver. Правильное написание сценариев требует разделения логики тестов и логики страниц, использования надежных локаторов и корректной синхронизации с интерфейсом.

## 3. Язык XPath. Основные конструкции, системные функции, работа с множествами элементов

### 3.1. Что такое XPath

XPath - это язык навигации по дереву XML/HTML-документа.  
В тестировании он используется для поиска элементов на веб-странице.

XPath особенно полезен, когда:
- элементы генерируются динамически;
- `id` нестабилен;
- нужно искать элемент по структуре DOM, тексту или комбинации признаков.

### 3.2. Почему XPath важен в этой работе

В условии лабораторной работы прямо указано, что сайт использует динамическую генерацию элементов, а значит выбор элемента должен строиться не по `id`, а с помощью `XPath`.

Это означает, что XPath здесь не просто вспомогательный инструмент, а основной способ адресации элементов.

### 3.3. Абсолютный и относительный XPath

#### Абсолютный XPath

Идет от корня документа:

```xpath
/html/body/div[2]/form/input
```

Недостатки:
- длинный;
- плохо читается;
- легко ломается при изменении DOM.

#### Относительный XPath

Идет от любого подходящего узла:

```xpath
//input[@name='login']
```

Преимущества:
- короче;
- понятнее;
- устойчивее к изменению структуры страницы.

Для Selenium почти всегда предпочтителен именно относительный XPath.

### 3.4. Базовые конструкции XPath

#### Выбор по имени тега

```xpath
//input
//button
//a
```

#### Выбор по атрибуту

```xpath
//input[@name='login']
//button[@type='submit']
```

#### Выбор по тексту

```xpath
//button[text()='Sign in']
```

#### Универсальный селектор `*`

```xpath
//*[@name='login']
```

Он означает «любой элемент», если остальные условия совпадают.

### 3.5. Разница между `/` и `//`

#### `/`
Выбирает только непосредственного потомка:

```xpath
//div/input
```

#### `//`
Ищет на любом уровне вложенности:

```xpath
//div//input
```

Это одно из важнейших различий в XPath.

### 3.6. Предикаты `[]`

Предикаты используются для фильтрации элементов.

Примеры:

```xpath
//input[@type='text']
//button[text()='Save']
//div[2]
```

В предикатах можно комбинировать условия.

### 3.7. Логические операции

#### `and`

```xpath
//input[@type='text' and @name='username']
```

#### `or`

```xpath
//input[@name='username' or @id='username']
```

Такие выражения позволяют делать локатор точнее и устойчивее.

### 3.8. Системные функции XPath

#### `contains()`

Позволяет искать частичное совпадение.

```xpath
//button[contains(text(),'Sign')]
//div[contains(@class,'header')]
```

Очень полезна, если значение атрибута длинное или содержит динамическую часть.

#### `starts-with()`

Проверяет, начинается ли строка с определенного префикса.

```xpath
//input[starts-with(@id,'user')]
```

#### `text()`

Позволяет обратиться к текстовому содержимому узла.

```xpath
//a[text()='Profile']
```

#### `normalize-space()`

Убирает лишние пробелы и позволяет устойчиво сравнивать текст.

```xpath
//button[normalize-space()='Sign in']
```

Это особенно полезно, когда в HTML есть переносы строк или лишние пробелы.

### 3.9. Работа с индексами

XPath индексирует элементы с единицы, а не с нуля.

```xpath
(//button[@type='submit'])[1]
(//button[@type='submit'])[2]
```

Это важно помнить, потому что в Java и в большинстве языков индексация обычно начинается с нуля.

### 3.10. Навигация по DOM и оси XPath

XPath поддерживает навигацию не только вниз по дереву, но и по отношениям между узлами.

#### Родитель

```xpath
//input[@name='login']/..
//input[@name='login']/parent::div
```

#### Предок

```xpath
//input[@name='login']/ancestor::form
```

#### Потомок

```xpath
//form/descendant::input
```

#### Следующий сосед

```xpath
//label[text()='Email']/following-sibling::input
```

#### Предыдущий сосед

```xpath
//input[@id='email']/preceding-sibling::label
```

Эти конструкции полезны, если у элемента нет хорошего собственного атрибута, но его можно найти через соседний или родительский узел.

### 3.11. Работа с множествами элементов

XPath может возвращать не только один элемент, но и набор элементов.

Пример:

```xpath
//div[@class='item']
```

В Selenium это часто используется с `findElements(...)`, когда нужно:
- получить список строк таблицы;
- найти все кнопки;
- проверить количество карточек на странице;
- выбрать первый или нужный элемент из множества.

Например:

```java
List<WebElement> rows = driver.findElements(By.xpath("//tr"));
```

Если нужно обратиться к конкретному элементу множества прямо в XPath:

```xpath
(//tr)[1]
(//tr)[2]
```

### 3.12. Практические рекомендации по XPath

При написании локаторов желательно:
- использовать относительный XPath;
- избегать абсолютных путей;
- опираться на устойчивые атрибуты;
- комбинировать признаки через `and`;
- использовать `contains()` и `normalize-space()` для динамического текста;
- не полагаться на хрупкие индексы, если можно обойтись без них.

### 3.13. Краткий вывод

XPath - это мощный язык поиска элементов в DOM. Его сила в том, что он позволяет находить элементы по тегам, атрибутам, тексту, структуре документа и отношениям между узлами. Для данной лабораторной работы XPath особенно важен, потому что по условию он является основным способом выбора элементов на динамических страницах.

---

## Итог

Для защиты лабораторной работы важно показать, что:

1. Функциональное тестирование строится от пользовательских use case и проверяет корректность работы функций.
2. Selenium позволяет автоматизировать веб-сценарии через архитектуру `тест -> WebDriver -> драйвер браузера -> браузер`.
3. XPath нужен для надежного поиска элементов на динамических страницах и поддерживает как базовые, так и продвинутые конструкции.

Если отвечать устно кратко, то лучше держать структуру ответа именно в таком порядке:  
`определение -> основные понятия -> практическое применение в данном проекте`.
